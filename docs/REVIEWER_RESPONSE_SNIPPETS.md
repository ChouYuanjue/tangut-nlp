# Reviewer Response Snippets

## Portability Validation

We agree that portability should be demonstrated empirically rather than claimed rhetorically. We therefore added a supplementary second-script stress test built from real OBIMD oracle-bone reading sequences canonicalized into the EVOBC ID inventory. The goal of this supplement is deliberately narrow: it tests whether the symbolic interface and local-training recipe transfer beyond Tangut without retuning the Tangut-specific workflow modules. Under this conservative setup, dictionary-grounded prompting is the strongest single oracle model (`40/200` exact; `46.20` chrF++), but the local MT-SFT model remains complementary rather than redundant. A simple open 2-way selector over the prompt and MT-SFT outputs improves the result further to `44/200` exact and `47.81` chrF++ while reducing contamination from `3.5%` to `0.5%`. We therefore frame the portability result as evidence that the workflow-level complementarity story transfers across scripts, not as a claim that Tangut-specific rerankers or evaluation modules transfer unchanged.

## Why We Do Not Add Expert Evaluation Yet

We do not treat the current paper as a scholar-facing deployment study. Its immediate purpose is to answer a narrower benchmark question under matched references: once a strong frontier prompt baseline is already in place, what local supervision remains useful, and can complementary frontier/local failures be converted into a better workflow? A lightweight expert panel at this stage would answer a different question, namely downstream scholarly utility, while still being too small to support strong claims about expert consensus. We therefore reserve expert evaluation for the next stage, after the candidate workflow family has already been narrowed technically and after the cross-script stress test has been established.

## Contamination Definition

We agree that the contamination metric should be operationalized explicitly rather than left impressionistic. In the revision we therefore define contamination as a deterministic binary artifact flag with no tuned threshold: an output is marked contaminated iff it contains any Latin alphabetic character, angle-bracket markup, an explicit `[UNK]` token, or one of the small prompt-artifact lexemes actually observed in the legacy runs (e.g., `assistant`, `manuals`, `networks`, `shows`). We further add a lightweight failure taxonomy over stored predictions that separates contamination from truncation, over-expansion, title-form slips (suffix drop / numeral mismatch), and residual content drift. This makes the narrative more concrete: frontier prompting is already clean, legacy DPO is the only representative system with material artifact leakage, and the open selector preserves the clean frontier regime while removing the one frontier truncation case.

## Closed-Judge Robustness

We agree that the closed reference-aware judge should not stand alone. The repository therefore includes a fully local Qwen2.5-7B surrogate judge (`scripts/run_local_reference_eval_suite.py`) over exactly the same source/reference/candidate triples. The local judge is more score-compressed than the Azure-backed judge, but it preserves the broad ranking used in the paper. For the local learned family, the surrogate ordering remains `final_v2 < baseline3_1_unk < baseline3_2_multitask < gap-filtered MT-DPO` (`4.04 < 4.06 < 4.08 < 4.10` on mean overall). For the workflow comparison, it preserves `frontier/open selector < closed hybrid < human reference` (`4.20 = 4.20 < 4.24 < 4.90`). We therefore use the Azure judge as the sharper evaluator, but not as the only evidence source.

## DPO Reward Transparency

We agree that the DPO reward should be specified more concretely. In the revision we now state explicitly that the reward uses lexical coverage plus a weak unnormalized `0.01 * log PPL` tie-breaker, where PPL is provided by a local Qwen2.5-0.5B causal LM. We also added a diagnostic reward-design ablation over all 3,867 existing preference pairs. Two results matter most. First, removing the PPL term entirely causes zero pair-order flips and leaves the retained-pair quality essentially unchanged (`81.5% -> 81.8%` chosen-better rate at gap `>= 0.4`), so the current pipeline is functionally lexical-coverage-driven. Second, replacing part of the reward with multitask `<literal>` overlap yields much cleaner retained pairs (`85.1%` good at gap `>= 0.2`) but at the cost of a much smaller usable pool (`832` vs `1996` pairs). We therefore keep the current DPO claim conservative: aggressive gap filtering is the main stabilizer for the present reward, while stronger structure-aware rewards remain future work rather than an unreported hidden ingredient.

## Frontier Prompt Robustness

We agree that the frontier baseline should be specified more concretely. The revision now enumerates the exact setup in the main text: `deepseek/deepseek-v3.2` via OpenRouter, dictionary-grounded prompt, four fixed development-set exemplars, temperature `0.0`, and maximum `512` output tokens. Because decoding is deterministic, sampling-seed variance does not apply here. We do not present a full paid prompt sweep as if it were free, but we do report the stored pilot prompt variants already in the repository. On the shared 5-item pilot slice, the final few-shot title-only prompt reaches `2/5` exact matches and `67.68` chrF++, compared with `0/5` and `21.99` chrF++ for the strict title-only prompt. The pilot therefore supports the paper's chosen prompt direction without pretending that prompt sensitivity is fully exhausted.
