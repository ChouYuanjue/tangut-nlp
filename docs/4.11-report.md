# 西夏文短标题翻译项目周报（2026-04-11）

## 1. 本周摘要

本周主要完成了三项工作：补齐强 frontier prompt-only 基线，回溯并修复 legacy DPO 的失稳原因，以及据此重新收束任务定义。基于当前证据，项目更适合表述为“超低资源历史短标题翻译”，而不是通用西夏文机器翻译。

当前可以稳定支持的判断如下：

- 强 prompt 的 frontier 中文模型已经给出了当前单模型上界，`frontier_deepseek_v32_fewshot_cot` 在 `50` 条测试集上达到 reference-aware overall `2.80/5`、exact match `12/50`、标题后缀保持 `24/27`、污染 `0/50`。
- 在开放、可训练范式内，`baseline3_2_multitask` 是当前最稳的 pure SFT 底座，`final_gap04_multitask_sigmoid` 是当前最强本地可训练单模型。
- 当前最强整体结果不是单个模型，而是结合 frontier 与本地保守候选的 workflow：`hybrid_multi3_catalog_gpt54` 达到 chrF++ `34.87`、reference-aware overall `3.02/5`、exact match `14/50`、污染 `0/50`。
- `human_reference` 的 stress test 继续表明，lexical coverage 和现代中文 LM perplexity 不能作为该任务的主结论指标。

| 层级 | 方法 | 当前结果 |
|---|---|---|
| 最强单模型（综合质量） | `frontier_deepseek_v32_fewshot_cot` | ref-aware overall `2.80/5`，exact match `12/50`，标题后缀保持 `24/27`，污染 `0/50` |
| 最强本地可训练单模型 | `final_gap04_multitask_sigmoid` | chrF++ `30.01`，ref-aware overall `2.22/5`，exact match `5/50`，长度比 `1.12` |
| 最强整体 workflow | `hybrid_multi3_catalog_gpt54` | chrF++ `34.87`，ref-aware overall `3.02/5`，exact match `14/50`，标题后缀保持 `26/27`，污染 `0/50` |

## 2. 本周完成

### 2.1 补齐强 frontier prompt-only 基线

针对“是否直接用更强中文大模型 + 更强提示词即可”这一问题，本周补做了强 frontier 基线。配置如下：

- 模型：`deepseek/deepseek-v3.2`
- 输入：西夏文原文 + 逐段词典 gloss
- 提示方式：严格标题体约束 + anti-expansion 约束 + few-shot
- few-shot 示例：`佛説大人八覺經`、`佛説菩薩行修經`、`金光明總持經`、`摩訶般若波羅蜜多心經`
- 解码：`temperature=0.0`

当前 prompt 的核心约束如下：

```text
系统提示：
- 任务是把西夏文短文本还原成尽可能准确、紧凑的现代中文标题
- 优先输出标题/书名体，而不是解释性句子
- 只能依据原文和词典释义判断，不能虚构上下文
- 多义词必须全局消歧，宁可保守，也不要把多个释义拼成长句
- 遇到泛化佛经名时，不要机械照抄成最终题名
- 优先保持 經 / 論 / 記 / 疏 / 頌 / 儀 / 義 / 傳 / 錄 等标题后缀
- 最终输出尽量短，只输出标题本身

用户提示：
- 给出西夏文原文
- 给出逐段词典释义
- 明确要求先识别专名、音译佛典词、结构词、题名后缀
- 再做全局消歧，避免把字面义累加成解释性句子
- 多候选时优先选择更像真实中文书名且更短的一项
```

结果如下：

| 方法 | chrF++ | Ref-aware Overall | Exact Match | 标题后缀保持 | 污染 |
|---|---:|---:|---:|---:|---:|
| `baseline2` | 6.13 | 1.50 | 0/50 | 3/27 | 0/50 |
| `baseline2_1_cot` | 7.59 | - | 1/50 | 3/27 | 5/50 |
| `frontier_deepseek_v32_fewshot_cot` | **28.54** | **2.80** | **12/50** | **24/27** | **0/50** |

这一结果说明，强 prompt 已经实做完成，而且当前确实是最强单模型。后续写作不能再把 prompt-only 当弱基线处理。

### 2.2 回溯 legacy DPO 失稳原因并完成修复

本周对旧版 DPO 管线做了回溯，问题现在比较清楚：

1. 底座选错。旧 selector 只在 `mixed / unk / semantic` 三个 SFT 底座中选最佳模型，直接排除了当前最强的 `multitask`。
2. 偏好对质量不足。原始 `3867` 对样本里包含 non-finite reward 和无效 pair。
3. reward gap 过松。低 gap `(0.05, 0.10]` 区间的 chosen-better rate 只有 `62.3%`，说明不少 pair 的偏好方向本身就不稳。
4. reward 与任务目标不完全一致，容易把模型往“词典命中提高，但标题越写越长”的方向推。

legacy DPO 的结果可以看出是明显失稳，而不是轻微回退：

| 模型 | chrF++ | Ref-aware Overall | Exact Match | 污染 | Length Ratio |
|---|---:|---:|---:|---:|---:|
| `baseline3_1_unk` | 22.11 | 1.82 | 3/50 | 3/50 | 1.05 |
| `baseline3_2_multitask` | 25.99 | 1.98 | 5/50 | 1/50 | 1.10 |
| `final_v2` | 19.39 | 1.68 | 2/50 | 9/50 | 1.88 |
| `final` | 9.23 | - | 0/50 | 17/50 | 4.74 |

修复后的流程是：

- 底座切到 `baseline3_2_multitask`
- 对偏好对先做有效性过滤、去重和 gap filter
- 分别在 `gap >= 0.2` 和 `gap >= 0.4` 上训练，重点看高 gap 版本

偏好对质量审计结果：

- 原始偏好对数：`3867`
- `gap >= 0.2` 保留：`1996`
- `gap >= 0.4` 保留：`498`
- 低 gap `(0.05, 0.10]` 的 chosen-better rate：`62.3%`
- `gap >= 0.4` 的 chosen-better rate：`81.5%`

修复后的关键结果：

| 模型 | chrF++ | Ref-aware Overall | Exact Match | 污染 | Length Ratio |
|---|---:|---:|---:|---:|---:|
| `baseline3_2_multitask` | 25.99 | 1.98 | 5/50 | 1/50 | 1.10 |
| `final_gap04_multitask_sigmoid` | **30.01** | **2.22** | **5/50** | 2/50 | **1.12** |
| `final_gap04_multitask_robustwpo` | 27.52 | **2.22** | **5/50** | 2/50 | 1.26 |

当前更准确的结论不是“DPO 天然无效”，而是：旧版 DPO 会崩；以 `multitask` 为底座并做高 gap 过滤后，DPO 才变成可用方法。

### 2.3 验证 frontier 与本地模型的互补性

本周继续核对了 workflow 层面的结果。当前最强结果不是单个模型直接生成，而是 frontier 候选与本地保守候选结合后的选择流程。

| 工作流 | chrF++ | Ref-aware Overall | Exact Match |
|---|---:|---:|---:|
| `frontier_deepseek_v32_fewshot_cot` | 28.54 | 2.80 | 12/50 |
| `open_hybrid_heuristic_guarded` | 32.62 | 2.90 | 12/50 |
| `hybrid_multi3_catalog_gpt54` | **34.87** | **3.02** | **14/50** |

这一增益不是靠自由扩写换来的：

- `hybrid_multi3_catalog_gpt54` 污染 `0/50`
- 标题后缀保持 `26/27`
- 长度比 `0.95`

这说明本地训练的价值不是去和 frontier 单模型直接比较绝对优劣，而是提供另一类更保守的候选，和 frontier 形成互补。

### 2.4 完成评估层 sanity check

`human_reference` 的 stress test 结果没有变化，但其结论仍然重要：

- `chrF++ = 100.00`
- `lexical coverage = 0.5733`
- `perplexity = 1646139.34`

这意味着在当前任务里：

- lexical coverage 不能直接代表翻译质量；
- 现代中文 LM 的 PPL 会系统性惩罚历史标题体；
- reference-free 指标不适合做主结论的 model selection signal。

## 3. 两个主要问题

### 3.1 之前的 DPO 为什么会失稳，当前是如何修复的

旧版 DPO 失稳，核心原因不是“DPO 这个方法本身不行”，而是训练输入同时存在三类错配：底座选错、偏好对太脏、gap 太松。旧 selector 没把 `multitask` 纳入候选，导致 `final_v2` 不是从当前最强 SFT 底座出发；原始偏好对里又混有 non-finite 和低置信样本，低 gap 区间的偏好方向只有 `62.3%` 的正确率，直接把噪声送进训练；再叠加 reward 与“短标题、少扩写”的真实目标不完全一致，最终就把模型推向了长度失控和污染上升。`final_v2` 的长度比达到 `1.88`、污染 `9/50`，`final` 更严重，长度比 `4.74`、污染 `17/50`。

当前修复的思路是先把训练信号做干净，再讨论 DPO 是否成立。具体做法是：改用 `baseline3_2_multitask` 作为底座；对 `3867` 对偏好样本先做有效性过滤和去重；只保留 `gap >= 0.4` 的 `498` 对高置信 pair 进入训练。在这一设置下，`final_gap04_multitask_sigmoid` 提升到 chrF++ `30.01`、ref-aware overall `2.22`、长度比 `1.12`、污染 `2/50`。因此当前可以向外说明的是：legacy DPO 确实训崩了，但原因已经定位并修复；修复后的 DPO 不是全面最强，却已经成为当前最强的本地可训练单模型。

### 3.2 为什么不直接用 SOTA 中文大模型加足够强的提示词

该对照已经完成，结果表明强 prompt frontier 模型确实是当前最强单模型。当前的 `frontier_deepseek_v32_fewshot_cot` 使用逐段词典 gloss、严格标题体约束、anti-expansion 约束、固定 `4` 个 few-shot、`temperature=0.0`，并明确要求先识别专名、音译佛典词、题名后缀，再做全局消歧，最后只输出尽量短的标题。在这一设置下，单模型结果已经达到 ref-aware overall `2.80/5`、exact match `12/50`、标题后缀保持 `24/27`、污染 `0/50`。

因此，后续写作不能再把问题表述为“强 prompt 仍然不够”。更准确的判断是：强 frontier prompt 给出了当前单模型上界；本地结构化训练的价值在于提供另一类更保守的候选，并在 workflow 层面与 frontier 形成互补。这也是当前最佳结果来自 `hybrid_multi3_catalog_gpt54`，而不是任一单模型的原因。

几个典型例子如下：

| 参考 | Frontier 输出 | 更保守的本地/工作流输出 |
|---|---|---|
| `五部經` | `五類經` | `五部經` |
| `番言金剛王乘根` | `番` | `番言金剛王乘根` |
| `正理滴之句義顯具` | `德宜點義句顯宣論` | `正理滴之句義顯了` |

## 4. 当前边界

- 真实平行数据只有 `491` 条，划分为 `400/41/50`，现阶段只能支撑超低资源结论。
- 测试集以经名、书名、目录项等短标题为主，不能外推为通用句子级西夏文翻译。
- 当前最强整体结果依赖 hybrid workflow，而不是单个开放模型。
- 评估上仍需把 chrF++、exact match、污染率、标题后缀保持和 reference-aware judge 放在主结论；lexical coverage、PPL 只能保留为诊断项。
